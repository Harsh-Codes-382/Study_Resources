## 1. What is EC2? (Virtual Machines in the Cloud)

![Amazon EC2 (Elastic Compute Cloud) overview — EC2 lets you launch Virtual Machines (VMs); a VM is a software emulation of a physical computer; multiple VMs run on the same physical server; a launched VM is called an "instance"; EC2 is highly configurable (CPU, RAM, network, OS, storage via EBS) chosen through an AMI; EC2 is the backbone of AWS](AWS_NOtes_Images/EC2_Intro.png)

**Elastic Compute Cloud (EC2)** allows you to launch **Virtual Machines (VMs)** in the AWS cloud.

### What is a Virtual Machine (VM)?
- A VM is an **emulation of a physical computer using software** — server *virtualization* lets you easily **create, copy, resize, or migrate** a server.
- **Multiple VMs run on the same physical server** so you can share the cost with other customers (AWS owns the physical box; you rent a slice).
- Think of a VM as if your computer were just an **executable file on another computer**.

### Key vocabulary
- When you launch a virtual machine, AWS calls it an **"instance."** (Exam loves this word — *instance = a running EC2 VM*.)
- An **AMI (Amazon Machine Image)** is a **predefined configuration / template for a virtual machine** — it bakes in the OS and software so a new instance boots ready-to-go.

### EC2 is a *highly configurable server*
When you launch an instance you choose an **AMI**, which affects options such as:
- **The amount of CPUs** (vCPUs).
- **The amount of Memory (RAM).**
- **The amount of Network Bandwidth.**
- **The Operating System (OS)** — e.g. Windows 10, **Ubuntu**, **Amazon Linux 2**.
- **Attach virtual hard-drives for storage** — e.g. **Elastic Block Store (EBS)**.

> **EC2 is considered the backbone of AWS** because the **majority of AWS services run on EC2 as their underlying servers** — e.g. **S3, RDS, DynamoDB, Lambda** are all built on top of EC2 behind the scenes. This is *why* EC2 is so heavily examined.

## 2. EC2 Instance Families

![EC2 Instance Families — instance families are different combinations of CPU, Memory, Storage and Networking capacity; the five families are General Purpose, Compute Optimized, Memory Optimized, Accelerated Optimized (GPU), and Storage Optimized, each with example instance types and target use-cases](AWS_NOtes_Images/EC2_Families.png)

### What are Instance Families?
**Instance families are different combinations of CPU, Memory, Storage and Networking capacity.** They let you choose the **appropriate combination of capacity to meet your application's unique requirements** — different families exist because the underlying **hardware varies** to give each its strength.

> **The mental model:** pick the family by **which resource your workload is hungry for.** Balanced? General Purpose. CPU-heavy? Compute Optimized. RAM-heavy? Memory Optimized. GPU/accelerator? Accelerated. Disk-I/O-heavy? Storage Optimized.

### The 5 EC2 Instance Families

| Family | Optimized for | Example instance types | Typical use-cases |
|---|---|---|---|
| **General Purpose** | A **balance** of compute, memory and networking | `A1, T2, T3, T3a, M4, M5, M5a, M5n, M6zn, M6g, M6i, Mac` | **Web servers, code repositories** |
| **Compute Optimized** | **High-performance processors** (CPU-bound) | `C5, C4, C5a, C5n, C6g, C6gn` | **Scientific modeling, dedicated gaming servers, ad-server engines** |
| **Memory Optimized** | Fast performance for workloads that process **large data sets in memory (RAM)** | `R4, R5, R5a, R5b, R5n, X1, X1e, High Memory, z1d` | **In-memory caches, in-memory databases, real-time big-data analytics** |
| **Accelerated Optimized** | **Hardware accelerators / co-processors** (GPUs, FPGAs) | `P2, P3, P4, G3, G4ad, G4dn, F1, Inf1, VT1` | **Machine learning, computational finance, seismic analysis, speech recognition** |
| **Storage Optimized** | **High sequential read/write** to **very large local data sets** (disk I/O) | `I3, I3en, D2, D3, D3en, H1` | **NoSQL, in-memory or transactional databases, data warehousing** |

### How to read the instance type names (handy for elimination)
- The **letter prefix tells you the family** — **`T`/`M`/`A` = General Purpose**, **`C` = Compute**, **`R`/`X`/`z` = memoRy (Memory Optimized)**, **`P`/`G`/`F`/`Inf` = accelerated (GPU/FPGA)**, **`I`/`D`/`H` = storage (I/O & Disk)**.
- The **number** is the **generation** (higher = newer, e.g. `M6` newer than `M5`).
- **`Mac`** instances run **macOS** (for building/testing Apple apps).

## 3. EC2 Instance Type Naming & Sizes (nano, micro, medium…)

Section 2 covered the **family** (which hardware). The other half of an instance type is its **size** — how *much* of that hardware you get. Together they form the full **instance type name** you actually pick when launching.

### How to read a full instance type name: `t3.micro`

```
   t          3          .          micro
   │          │                       │
 Family   Generation              Size
(General  (3rd gen,            (how much CPU/RAM
 Purpose)  newer = better)      in this instance)
```

| Part | `t3.micro` | Meaning |
|---|---|---|
| **Family letter** | `t` | The instance family — General Purpose (see §2) |
| **Generation number** | `3` | Hardware generation; **higher = newer/better** (`t3` newer than `t2`) |
| **Size** | `micro` | The capacity tier — roughly *doubles* as you go up the ladder |

> Some names add extra letters after the generation (e.g. `m5**a**`, `c5**n**`, `r5**b**`) indicating a variant — `a` = AMD CPU, `n` = network-enhanced, `g` = AWS Graviton (ARM). For the exam you mainly need to recognise **family + generation + size**.

### The size ladder (smallest → largest)
As you climb each step, **vCPU and memory roughly double** (and so does the price). The common sizes:

| Size | Relative capacity | Typical fit |
|---|---|---|
| **nano** | Smallest, tiny RAM | Tiniest workloads, learning, micro-services |
| **micro** | Very small (**`t2.micro` / `t3.micro` are Free Tier eligible**) | Low-traffic test/dev, free-tier practice |
| **small** | Small | Light apps, small websites |
| **medium** | Modest | Small production apps, dev environments |
| **large** | Baseline "real" size | General workloads, small databases |
| **xlarge** | ~2× large | Busier apps |
| **2xlarge, 4xlarge, 8xlarge … 16xlarge, 24xlarge, etc.** | Each step roughly doubles | Heavy/enterprise workloads |
| **metal** | The whole **bare-metal** physical server (no hypervisor) | Workloads needing direct hardware access |

> **Rule of thumb:** going **nano → micro → small → medium → large → xlarge → 2xlarge…** each tier **roughly doubles the vCPUs, the RAM, and the cost.** Pick the **smallest size that meets your needs** to control cost (then scale up — "vertical scaling" — or scale out with more instances — "horizontal scaling," via Auto Scaling, see [[05_Computing_Services]]).

> **Free Tier exam fact:** **`t2.micro`** (and in many regions `t3.micro`) is the **EC2 Free Tier** instance — 750 hours/month for the first 12 months. If a question mentions "free tier" + EC2, think **`t.micro`**.

## 4. EC2 Tenancy — Dedicated Hosts vs Dedicated Instances (+ BYOL)

**Tenancy** = *how the underlying physical server is shared.* By default your EC2 instances share hardware with other AWS customers; for **isolation, compliance, or licensing** reasons you can pay to **not** share. The exam tests the **three tenancy options** and *why* you'd pick each.

### The three tenancy options

| Tenancy | Who shares the physical server? | Cost | Why choose it |
|---|---|---|---|
| **Shared (default)** | **Multiple customers** share the same physical host (multi-tenant), isolated by the hypervisor | **Cheapest** | The normal default — fine for almost everything |
| **Dedicated Instance** | **Only you** (single-tenant) — your instances run on hardware dedicated to your account | More expensive | **Regulatory/compliance** needs "no shared hardware," but you don't care *which* physical box |
| **Dedicated Host** | **Only you**, *and* you get a **whole physical server** that you can see and control | **Most expensive** | **BYOL** (per-socket/per-core licenses), strict compliance, control over physical placement |

> Both **Dedicated Instances** and **Dedicated Hosts** give you **single-tenant** (no other customer on your hardware). The difference is **how much visibility and control** you get over the *physical* server underneath.

### Dedicated Instance vs Dedicated Host (the comparison the exam wants)

| Feature | **Dedicated Instance** | **Dedicated Host** |
|---|---|---|
| **Hardware isolation** | Single-tenant (dedicated to your **account**) | Single-tenant — you get the **entire physical server** |
| **Visibility of the physical host** | **No** — you can't see sockets/cores/host ID | **Yes** — you see **sockets, physical cores, and Host ID** |
| **Control over instance placement** | **No** — instances may land on different hosts | **Yes** — you control placement (**host affinity**: keep an instance on the *same* host) |
| **BYOL** with per-socket / per-core / per-VM licenses | **Not supported** (no host visibility) | **Supported** — the whole point of Dedicated Hosts |
| **Billing** | Per **instance** | Per **host** (you pay for the whole server, regardless of how many instances you run on it) |
| **Typical reason** | "Compliance says **no shared hardware**" | "I must reuse **existing server-bound licenses**" or "regulator requires a **specific physical host**" |

### BYOL — Bring Your Own License (the killer reason for Dedicated Hosts)

- **BYOL** means reusing software licenses you **already own** (e.g. **Windows Server, SQL Server, Oracle, SUSE/RHEL**) on AWS instead of paying AWS to include the license.
- Many of these licenses are **bound to physical hardware** — priced **per physical socket or per physical core**.
- To honor such a license you must **see and control the actual sockets/cores** — which **only a Dedicated Host provides.**
- ➡️ **Exam shortcut:** the moment a question says **"bring your own license," "existing licenses," "per-socket / per-core licensing,"** the answer is **Dedicated Host** (not Dedicated Instance, not Shared).

> **One-line memory hook:** **Dedicated *Instance* = isolated hardware** (compliance: "no neighbors"). **Dedicated *Host* = your *own physical server*** you can see and control (BYOL + placement). If the trigger is **license**, pick **Host**; if it's just **"no shared tenancy,"** an Instance is enough.

## 5. Elastic IP (EIP) — Static Public IP Addresses

By default, when you stop and start an EC2 instance its **public IPv4 address changes** (AWS hands it a new one from a shared pool). That breaks anything pointing at the old address — DNS records, whitelists, bookmarks. An **Elastic IP** solves this: it's a **static, public IPv4 address you own in your account** and can **attach to any instance** (or move between instances).

### Key facts

- An **Elastic IP is a static public IPv4 address** that **stays the same** even when you stop/start the instance.
- You can **remap** (re-associate) an EIP from a failed instance to a healthy one — useful for **quick failover** without changing DNS.
- An EIP is **associated with your AWS account** (per Region) until you choose to release it.
- **NAT Gateways** also use an Elastic IP to give private-subnet instances a fixed outbound address (see [[08_Networks]]).

### 💰 Billing gotcha (very commonly tested)

> **An Elastic IP is FREE *only while it is attached to a running instance*.** You are **charged for an EIP that is allocated but NOT in use** — i.e. not associated with a running instance (idle/unattached, or attached to a stopped instance). AWS does this to discourage hoarding scarce IPv4 addresses.

- ➡️ **Exam shortcut:** "Why am I being billed for an Elastic IP?" → because it's **allocated but not associated with a running instance.** Fix = attach it or **release** it.

> **Memory hook:** *Elastic IP = a fixed phone number for your server.* Free while plugged in; you pay a "reservation fee" if you reserve the number and let it sit idle.

## 6. AMI — Amazon Machine Image (Deep Dive)

You met the AMI in §1 as the **template** an instance is built from. Because it's so heavily tested, here's the detail you need.

### What an AMI contains
An **AMI (Amazon Machine Image)** is a **read-only template** that holds everything needed to launch an instance:
- The **Operating System** and its configuration.
- **Pre-installed software / applications** and settings.
- The **block-device mapping** (which EBS volumes/snapshots to attach at launch).
- Launch **permissions** (who can use the AMI).

> **The big idea:** bake a fully-configured server **once** into an AMI ("golden image"), then launch **many identical instances** from it in **seconds** — consistent, repeatable, fast. This is how Auto Scaling spins up new instances quickly (see [[05_Computing_Services]]).

### The 4 sources of AMIs (where they come from)

| AMI source | What it is | Cost |
|---|---|---|
| **Quick Start (AWS-provided)** | Default Amazon Linux, Ubuntu, Windows, etc. AMIs maintained by AWS | Free (pay only for the instance) |
| **AWS Marketplace** | **Third-party vendor** AMIs with software pre-baked (often licensed) | May carry a **software/licensing charge** on top of the instance |
| **My AMIs (Custom)** | AMIs **you created** from your own configured instances ("golden image") | Free (pay for the EBS snapshot storage) |
| **Community AMIs** | Shared publicly by other AWS users | Free (use with caution — not vetted by AWS) |

### AMIs are **Region-scoped**
- An AMI **lives in the Region where it was created.** To use it in another Region you must **copy the AMI** to that Region (this also enables cross-Region disaster recovery / deployments).
- AMIs are backed by **EBS snapshots** stored in **S3** behind the scenes — so a custom AMI's storage cost is really the snapshot cost.

> **AMI vs Snapshot:** an **EBS snapshot** is a point-in-time backup of **one volume**; an **AMI** is a full **launchable template** (it can reference *several* snapshots + OS + permissions). You launch an **instance** from an **AMI**, but you only **restore a volume** from a **snapshot**. (EBS & snapshots: [[06_Storage_Services]].)

## 7. Launch Templates (and Launch Configurations)

When you launch an instance you must specify a pile of settings — AMI, instance type, key pair, security groups, storage, user-data script, tags. A **Launch Template** lets you **save all of those settings once** and reuse them, so every instance launches **consistently** and you don't re-enter everything each time.

### Launch Template — what it does
- A **Launch Template stores the launch parameters** (AMI ID, instance type, key pair, security groups, EBS config, user data, tags, etc.) so you can **launch instances from it repeatedly.**
- **Supports versioning** — you can create multiple **versions** of a template (e.g. v1, v2, v3) and pick which one to launch from. Great for controlled, incremental changes.
- Used by **EC2 Auto Scaling** and the EC2 console/CLI to launch instances in a **standard, repeatable** way.

### Launch Template vs Launch Configuration (the exam comparison)

| | **Launch Template** ✅ (newer, recommended) | **Launch Configuration** (older, legacy) |
|---|---|---|
| **Versioning** | **Yes** — multiple versions | **No** — immutable, must recreate to change |
| **Feature support** | Full & current (T2/T3 unlimited, mixed instance types, Spot+On-Demand mix, etc.) | Limited / older features only |
| **Used by** | Auto Scaling **and** direct launches | Auto Scaling only |
| **AWS guidance** | **Recommended** going forward | **Legacy** — AWS steers you to Launch Templates |

> ➡️ **Exam shortcut:** if a question asks which to use for new Auto Scaling setups or mentions **versioning / latest features**, the answer is **Launch Template**. **Launch Configuration = the old, no-versioning one.**

> **Don't confuse the two "template" words:** an **AMI** is the *image* (OS + software baked into a disk). A **Launch Template** is the *form of launch settings* (which AMI, which instance type, which SG…). The Launch Template **references** an AMI; it doesn't replace it.

## 8. Elastic Load Balancing (ELB) & Load Balancer Types

Once you run **more than one instance** (see why in §5 — a single Elastic IP only points at one machine), you need something to sit **in front** of them and spread traffic across them. That's a **load balancer**, and AWS's managed version is **Elastic Load Balancing (ELB)**.

### What a load balancer actually does (the "why")

A load balancer is a **single, stable front door** that distributes incoming traffic across a pool of backend instances. It gives you four things a lone instance can't:

1. **Distributes load** — no single instance gets overwhelmed; traffic is spread across all healthy targets.
2. **High availability** — it runs across **multiple Availability Zones**, so if one AZ or instance dies, traffic keeps flowing to the survivors.
3. **Health checks** — it continuously pings each target; an instance that fails is **automatically taken out of rotation** (and put back when it recovers). Users never hit a dead server.
4. **Decoupling / stable endpoint** — clients only ever know the load balancer's address. Instances behind it can be added, removed, replaced, or scaled (by **Auto Scaling**) **without anyone outside ever noticing.** (See decoupling in [[02_Cloud_Architecture]] and Auto Scaling in [[05_Computing_Services]].)

> **ELB is a managed, auto-scaling service** — AWS runs it, patches it, and **scales its capacity automatically.** You never size or scale the ELB itself (see the scaling discussion: its IPs churn as it grows, which is why you advertise its **DNS name**, not an IP).

### The OSI layer idea (how to tell the types apart)

The load-balancer types differ by **which network "layer" they operate at** — i.e. *how deep they look into the traffic:*
- **Layer 7 (Application layer)** = understands **HTTP/HTTPS** — can read URLs, paths, headers, hostnames. **Smarter**, web-aware.
- **Layer 4 (Transport layer)** = sees only **TCP/UDP + IP + port** — doesn't read the content. **Faster**, dumber.
- **Layer 3 (Network layer)** = raw **IP packets** — used to route traffic through appliances.

### The 4 ELB types

| Type | Layer | Speaks | Best for | Fixed IP? |
|---|---|---|---|---|
| **ALB** — Application Load Balancer | **Layer 7** | HTTP / HTTPS / WebSocket | **Web apps, APIs, microservices, containers** — anything where you route by URL/host | ❌ DNS name only |
| **NLB** — Network Load Balancer | **Layer 4** | TCP / UDP / TLS | **Ultra-high performance, low latency, millions of req/s, non-HTTP, or when you need a static IP** | ✅ **Static / Elastic IP** per AZ |
| **GLB** — Gateway Load Balancer | **Layer 3** | IP packets | Routing traffic through **3rd-party virtual appliances** (firewalls, IDS/IPS, deep packet inspection) | n/a (transparent) |
| **CLB** — Classic Load Balancer | Layer 4 & 7 | HTTP/HTTPS/TCP | **Legacy** — the original ELB; **avoid for new work**, AWS steers you to ALB/NLB | ❌ |

#### ALB — Application Load Balancer (your default for web apps)
- Operates at **Layer 7**, so it can **read the request** and route by **content**: by **path** (`/api` → API servers, `/images` → image servers), by **hostname** (`shop.site.com` vs `blog.site.com`), by headers, query strings, HTTP method.
- Ideal for **HTTP/HTTPS apps, REST APIs, microservices, and containers** (integrates with ECS/EKS).
- Supports **SSL/TLS termination** (offload HTTPS decryption to the LB), **WebSockets**, and redirects.
- Gives you a **DNS name** (no static IP) → point your domain at it with a **CNAME** (or a Route 53 **Alias**).

#### NLB — Network Load Balancer (speed + static IP)
- Operates at **Layer 4** — just forwards TCP/UDP **blazingly fast** with **ultra-low latency**, handles **millions of requests per second**, and copes with **sudden spiky traffic**.
- The **only ELB that can have a static / Elastic IP** (one per AZ) → use it when a **3rd party must allow-list a fixed IP**, or for **non-HTTP** protocols (gaming, IoT, databases, real-time streams).
- Can preserve the **client's source IP**.

#### GLB — Gateway Load Balancer (security appliances)
- Layer 3. Lets you deploy and scale **3rd-party network appliances** (firewalls, intrusion detection, packet inspection). Traffic is transparently routed *through* the appliance fleet. Niche — know what it's *for* on the exam.

#### CLB — Classic Load Balancer (legacy)
- The **original** load balancer (Layer 4 + basic Layer 7). **Deprecated** for new designs — if a question contrasts "old vs new," CLB is the old one; pick **ALB/NLB** going forward.

### Core building blocks every dev should know

| Concept | What it is |
|---|---|
| **Listener** | A rule that checks for **connections on a port/protocol** (e.g. listen on HTTPS:443) and decides where to forward them |
| **Target Group** | The **pool of backends** the LB sends traffic to — EC2 instances, IP addresses, containers, or **Lambda** (ALB only). Health checks run **per target group** |
| **Health Check** | Periodic probe (e.g. `GET /health` returns 200) — unhealthy targets are **removed from rotation** automatically |
| **Routing / Listener Rules** (ALB) | Path-based & host-based rules that send different URLs to different target groups |
| **Sticky Sessions** (session affinity) | Pin a given user to the **same instance** (via a cookie) — needed when the app stores session state locally |
| **SSL/TLS Termination** | The LB decrypts HTTPS so your instances don't have to (offloads CPU); pairs with **ACM** for free managed certs |
| **Cross-Zone Load Balancing** | Spreads traffic **evenly across instances in all AZs**, not just within the same AZ |
| **Internet-facing vs Internal** | **Internet-facing** = public clients; **Internal** = only reachable inside the VPC (e.g. between microservice tiers) |
| **Connection Draining / Deregistration Delay** | Lets in-flight requests **finish gracefully** before an instance is removed |

### How it fits together (the whole picture)

```
                                       ┌──► [ Target Group ]──► Instance (AZ-a)  ✅ healthy
   Client ─► DNS name / Elastic IP ─►  │
            (the ELB = stable door)    ├──► [ Target Group ]──► Instance (AZ-b)  ✅ healthy
                                       │
                                       └──► [ Target Group ]──► Instance (AZ-c)  ❌ failed health check → removed

   Auto Scaling adds/removes instances in the target group automatically.
   ELB scales itself. Clients never see any of this churn.
```

> **The dev takeaway:** ELB + **Auto Scaling** + **multi-AZ** is the standard pattern for a **highly available, scalable** app. Auto Scaling handles *how many* instances; the ELB handles *spreading traffic and hiding failures*; multiple AZs handle *surviving a datacenter outage*.

## 9. Auto Scaling Groups (ASG)

A load balancer (§8) spreads traffic across instances — but **who decides how many instances exist, and replaces them when they die?** That's the **Auto Scaling Group (ASG)**. It's the piece that makes your fleet **elastic** (grow/shrink with demand) and **self-healing** (auto-replace failures).

### What an ASG does (the "why")
- **Automatic scaling** — adds instances when demand rises (**scale out**) and removes them when it falls (**scale in**), so you pay only for what you need.
- **Self-healing** — if an instance fails a health check, the ASG **terminates it and launches a fresh replacement** automatically. No human paged at 3 a.m.
- **High availability** — spreads instances **across multiple AZs**, and rebalances if one AZ goes down.
- **Elasticity** — the practical realisation of the *elasticity* pillar (see [[02_Cloud_Architecture]]) and AWS Auto Scaling overview (see [[05_Computing_Services]]).

### The three capacity numbers (exam loves these)

| Setting | Meaning |
|---|---|
| **Minimum** | The **floor** — ASG never drops below this many instances (your baseline availability) |
| **Desired (capacity)** | The number the ASG **tries to maintain right now** — scaling policies adjust this |
| **Maximum** | The **ceiling** — ASG never exceeds this (your cost guardrail) |

> If **Desired = 4** and an instance dies, the ASG sees only 3 running, notices it's **below desired**, and launches **1 replacement** to get back to 4. That single behaviour gives you **self-healing for free.**

### How the ASG knows what to launch
- It uses a **Launch Template** (see §7) — the saved recipe (which **AMI**, instance type, key pair, security groups, user data). So **§7 + §9 click together**: the Launch Template says *what each instance looks like*; the ASG says *how many and where*.

### Scaling policies (how it decides to add/remove)

| Policy | How it triggers | Example |
|---|---|---|
| **Target Tracking** ⭐ | Keep a metric at a target value (simplest, recommended) | "Keep **average CPU at 50%**" — ASG adds/removes instances to hold that |
| **Step Scaling** | Add/remove **different amounts** based on alarm severity | CPU > 70% → +1; CPU > 90% → +3 |
| **Simple Scaling** | One alarm → one fixed adjustment (older) | CPU > 80% → +1 |
| **Scheduled Scaling** | Scale at **known times** | Scale to 10 instances every weekday 9 a.m. |
| **Predictive Scaling** | ML **forecasts** demand and scales **ahead** of it | Pre-warm for a daily traffic spike |

> Scaling metrics come from **CloudWatch alarms** (e.g. CPUUtilization, request count). CloudWatch watches → triggers the policy → ASG changes **Desired** capacity. (CloudWatch: see [[03_Management_and_Development_Tools]].)

### ASG health checks — EC2 vs ELB (a subtle but tested distinction)
- **EC2 health check** (default) — only checks the instance is **running** at the hypervisor level. An instance can be "running" but the **app inside is crashed**.
- **ELB health check** — uses the **load balancer's** health check (e.g. `GET /health` returns 200) to judge health, so a hung app **is** caught and replaced.
- ➡️ **Best practice:** turn on **ELB health checks** on the ASG so a broken *application* (not just a broken VM) triggers replacement.

> **Cooldown / warm-up:** after a scaling action the ASG **waits** (cooldown) before scaling again, so it doesn't over-react to a brief spike (avoids "thrashing").

## 10. Connecting the Dots — Route 53 → ELB → Target Group → ASG → Instances

This is the **single most important architecture to internalise** for both the exam and real work. Here's how all the pieces from this topic fit into **one highly-available, auto-scaling web app**:

```
                       ① Route 53 (DNS)
                          "mysite.com?"  ──► returns the ELB's name/IP
                                │
                                ▼
                       ② Elastic Load Balancer (ALB/NLB)
                          • the single stable front door
                          • Listener (e.g. HTTPS:443) + rules
                                │  forwards to a...
                                ▼
                       ③ Target Group
                          • the registered pool of backends
                          • runs health checks on each target
                          │           │           │
            ┌─────────────┘           │           └─────────────┐
            ▼                         ▼                          ▼
   ④ Instance (AZ-a)        ④ Instance (AZ-b)         ④ Instance (AZ-c)
            ▲                         ▲                          ▲
            └─────────────┬──────────┴──────────────────────────┘
                          │ registers / replaces / scales
                       ⑤ Auto Scaling Group (ASG)
                          • launches instances from a Launch Template (§7)
                          • keeps Min ≤ Desired ≤ Max
                          • auto-registers new instances INTO the Target Group
                          • replaces unhealthy ones (self-healing)
```

### Who does what (the division of labour)

| Component | Its one job | What it does **not** do |
|---|---|---|
| **Route 53** | **DNS** — turns `mysite.com` into the load balancer's address; can do health checks & failover/latency/weighted routing | Doesn't balance per-request load across instances |
| **ELB (ALB/NLB)** | **Distributes each request** across healthy targets; the stable public endpoint | Doesn't create or destroy instances |
| **Target Group** | **The list of backends** the ELB sends to + **per-target health checks** | Doesn't decide *how many* instances exist |
| **ASG** | **How many** instances exist; **launches/replaces/scales** them; **registers** them into the target group | Doesn't route individual user requests |
| **Instances** | **Run your actual app** | Don't manage themselves |
| **Launch Template** | **The recipe** for each instance (AMI, type, SG…) | Isn't a running thing |

### The end-to-end request flow (say it out loud to lock it in)
1. A user's browser asks **Route 53**: *"where is `mysite.com`?"* → Route 53 returns the **ELB's** DNS name (ALB) or Elastic IP (NLB).
2. The request hits the **ELB's listener**, which forwards it to the **target group**.
3. The target group picks a **healthy instance** (skipping any failing its health check) and the **ELB** routes the request there.
4. Meanwhile the **ASG** is watching load via **CloudWatch**: traffic up → it **launches more instances** (from the **Launch Template**) and **auto-registers them into the target group**; traffic down → it **removes** some.
5. If any instance fails its health check, the **ASG terminates and replaces it**, and the **target group stops sending it traffic** — the user never notices.

### The mental hooks
- **Route 53 = the signpost** (find the front door). **ELB = the front door / receptionist** (spread people inside). **Target Group = the guest list + door bouncer** (who's allowed/alive). **ASG = the staffing manager** (hire/fire to match the crowd, replace no-shows). **Launch Template = the job description** (what every new hire looks like).
- **The glue:** the **ASG registers instances *into* the Target Group**, and the **ELB routes *to* that Target Group.** That registration link is *how* a freshly auto-scaled instance starts receiving traffic with zero manual steps.
- ⚠️ **You attach the ASG to a *Target Group*, not directly to the load balancer** (modern ALB/NLB way). The ELB then serves whatever is healthy in that target group.

> **One-line summary:** **Route 53** sends users to the **ELB**, which load-balances across healthy instances in a **Target Group**, which an **Auto Scaling Group** keeps correctly sized and self-healed using a **Launch Template** — together = a **highly available, elastic, self-healing** application.

## 11. EC2 Pricing Models (How You Pay for EC2)

![EC2 Pricing Models overview — there are 5 different ways to pay for EC2 virtual machines: On-Demand (least commitment, low cost and flexible, pay per hour/second, short-term spiky unpredictable workloads, cannot be interrupted, for first-time users); Spot (up to 90% off, biggest savings, request spare capacity, flexible start/end, can handle interruptions, non-critical background jobs); Reserved (up to 75% off, best long-term, steady predictable usage, 1 or 3 year commitment, can resell unused); Dedicated (most expensive, dedicated servers, on-demand or spot, guarantee of isolated hardware for enterprise requirements); plus AWS Savings Plan which can be used for more than just EC2](AWS_NOtes_Images/EC2_Pricing_Model.png)

Choosing the **right pricing model** is one of the biggest cost-optimisation levers on AWS (a Well-Architected pillar — see [[02_Cloud_Architecture]]). The exam tests **which model fits which workload pattern.** There are **5 ways to pay** for EC2 + the **Savings Plans** discount mechanism on top.

### The 5 ways to pay (the big picture)

| Model | Discount vs On-Demand | Commitment | Pick it when… |
|---|---|---|---|
| **On-Demand** | none (baseline) | **None** | Short-term, **spiky, unpredictable** workloads; new/experimental apps; can't be interrupted |
| **Reserved (RI)** | **up to 75%** | **1 or 3-year** | **Steady-state, predictable** usage; need **reserved capacity** |
| **Spot** | **up to 90%** | None (but **interruptible**) | **Fault-tolerant, flexible-timing**, non-critical batch/background jobs |
| **Dedicated** | — (**most expensive**) | varies | Need **isolated physical hardware** (compliance/licensing) — see Tenancy §4 |
| **Savings Plans** | up to **72%** | **1 or 3-year** ($/hr commit) | Predictable spend, want RI-like savings with **more flexibility** + simpler buying |

> **Mental model — commitment buys discount.** The **more you commit** (longer term / more upfront / accept interruption), the **bigger the discount.** On-Demand = zero commitment, zero discount. Spot = accept interruption → biggest discount. Reserved/Savings Plans = commit time → big discount.

---

### 11.1 On-Demand — Pay-As-You-Go (PAYG)

![EC2 On-Demand pricing — On-Demand is a Pay-As-You-Go (PAYG) model where you consume compute then pay; it is the default pricing when you launch an instance; no up-front payment and no long-term commitment; charged by the second (minimum of 60 seconds) for Linux/Windows/Windows-with-SQL editions without a separate hourly charge, or by the full hour for all other instance types; pricing is shown as an hourly rate; best for short-term, spikey or unpredictable workloads and for new app development or experiments](AWS_NOtes_Images/EC2_ONDemand.png)

- **Pay-As-You-Go (PAYG):** you consume compute, then pay — the **default** pricing when you launch an instance.
- **No up-front payment** and **no long-term commitment.**
- **Billing granularity:** charged **per-second (minimum 60 seconds)** for **Linux & Windows** (and Windows with SQL Standard/Enterprise/Web that have no separate hourly charge); **per full hour** for all other instance types.
- Pricing is always quoted as an **hourly rate.**
- ➡️ **Use for:** **short-term, spiky, unpredictable** workloads, **new apps under development**, or **experiments** — anywhere you can't predict usage and don't want to commit.

---

### 11.2 Reserved Instances (RI)

![EC2 Reserved Instances (RI) — designed for applications with steady-state, predictable usage or that require reserved capacity; reduced pricing is based on Term x Class Offering x RI Attributes x Payment Option; Term: 1-year or 3-year commitment (longer = greater savings), RIs do not renew automatically and revert to On-Demand with no interruption when they expire; Class: Standard up to 75% off (can modify RI attributes), Convertible up to 54% off (can exchange for an RI of equal or greater value), Scheduled is no longer offered by AWS; Payment Options: All Upfront (full payment at start, greatest savings), Partial Upfront (portion upfront plus discounted hourly), No Upfront (discounted hourly billed every hour regardless of use); RIs can be shared between multiple accounts within an AWS Organization, and unused RIs can be sold in the Reserved Instance Marketplace](AWS_NOtes_Images/EC2_Reserved_Instance.png)

For **steady-state, predictable** usage, or when you need **reserved capacity.** You commit to EC2 for a **1-year or 3-year** term in exchange for a big discount. **RIs do not auto-renew** — when one expires the instance simply reverts to **On-Demand pricing with no interruption.**

> **The price you get = `Term × Class × RI Attributes × Payment Option`.**

**Term (longer = greater savings):** **1 year** or **3 years.**

**Class (the flexibility ↔ discount trade-off):**
| Class | Max discount | Flexibility |
|---|---|---|
| **Standard** | **up to 75%** | Can **modify RI attributes** (AZ, size within family, etc.) — but **can't change instance family** |
| **Convertible** | **up to 54%** | Can **exchange** the RI for a different one of **equal or greater value** (change family/OS/tenancy) |
| **Scheduled** | — | **No longer offered** by AWS (legacy) |

**Payment Options (more upfront = greater savings):**
| Option | How you pay |
|---|---|
| **All Upfront** | **Full payment at the start** → biggest discount |
| **Partial Upfront** | A portion upfront; remaining hours billed at a **discounted hourly rate** |
| **No Upfront** | **Discounted hourly rate billed every hour**, regardless of whether you use it |

**Two more RI facts the exam likes:**
- **RIs can be shared between multiple accounts within an AWS Organization** (consolidated billing spreads the discount).
- **Unused RIs can be sold in the Reserved Instance Marketplace.**

#### Regional RI vs Zonal RI (RI Scope) — important detail

When you buy an RI you choose its **scope**: the whole **Region** or a specific **Availability Zone.** This changes what you actually get:

| | **Regional RI** | **Zonal RI** |
|---|---|---|
| **Scope** | Applies across **any AZ in the Region** | Locked to **one specific AZ** |
| **Capacity reservation** | ❌ **No** — billing discount only | ✅ **Yes** — **guarantees capacity** in that AZ |
| **Instance Size Flexibility** | ✅ **Yes** — discount auto-applies across **sizes in the same family** (Linux/Unix, default tenancy) via normalization | ❌ **No** — fixed AZ **and** size |
| **AZ flexibility** | ✅ Discount floats across AZs | ❌ None |

> ➡️ **Exam shortcut:** need a **guaranteed capacity reservation** in a specific AZ → **Zonal RI**. Want a **flexible billing discount** that floats across AZs and instance sizes → **Regional RI** (the more common choice). *(If you only need a capacity guarantee without the 1/3-yr commitment, use an **On-Demand Capacity Reservation** instead.)*

#### RI Limits (purchase limits)
- By **default**, you can purchase up to **20 Regional RIs per Region per month**, **plus 20 Zonal RIs per Availability Zone per month.**
- These are **soft limits** — you can request an increase via AWS Support. *(This is deeper than typical CLF-C02 scope, but good to recognise.)*

### 11.3 Spot Instances

![EC2 Spot Instances — Spot uses AWS's unused/spare compute capacity to maximise the utility of idle servers; like a hotel offering discounts to fill vacant suites or a plane filling vacant seats; provides up to 90% discount compared to On-Demand; Spot instances can be terminated if the capacity is needed by other On-Demand customers; designed for workloads with flexible start and end times or that are only feasible at very low compute cost (load balancing, flexible, and big-data/MapReduce workloads); AWS Batch is an easy and convenient way to use Spot pricing; Termination conditions: instances can be terminated by AWS at any time, if terminated by AWS you are not charged for a partial hour of usage, but if YOU terminate the instance you are still charged for any hour that it ran](AWS_NOtes_Images/EC2_Spot_Instance.png)

Spot sells AWS's **unused/spare compute capacity** at a steep discount (**up to 90% off**) to maximise the utility of idle servers. The catch: AWS can **reclaim it whenever it needs the capacity back.**

> **Analogy:** like a **hotel discounting vacant rooms** or an **airline discounting empty seats** — cheap, but only available while spare, and you might get bumped.

- ➡️ **Use for:** **fault-tolerant, flexible-timing** work — **batch jobs, big-data / MapReduce, CI/CD, background processing, load-balanced stateless workers.** **`AWS Batch`** is a convenient way to run on Spot.
- ❌ **Don't use for:** anything that **can't tolerate interruption** (databases, long single jobs that can't checkpoint).

**Termination conditions (commonly tested):**
- AWS can **terminate a Spot instance at any time** (with a ~2-minute warning).
- If **AWS terminates** it, you are **not charged for the partial hour.**
- If **you terminate** it yourself, you **are still charged for any hour it ran.**

---

### 11.4 Dedicated (Hosts / Instances)

The **most expensive** model — you pay for **isolated physical hardware** (no other customers on your server). It can be run **On-Demand or via Spot**, and is used when **compliance or licensing requires a guarantee of isolated hardware.**

> This is the **pricing/tenancy** view of what §4 already covered in depth — see **Tenancy §4** for **Dedicated Host vs Dedicated Instance** and the **BYOL** rule. **"Isolated hardware / BYOL / per-socket license" → Dedicated.**

### 11.5 AWS Savings Plans

![AWS Savings Plans (part 1) — Savings Plans offer similar discounts to Reserved Instances (RI) but simplify the purchasing process; there are 3 types: Compute Savings Plans, EC2 Instance Savings Plans, and SageMaker Savings Plans; you can choose one of two terms (1-year or 3-year); you commit to an hourly dollar amount (an hourly commitment); payment options are All Upfront, Partial Upfront, or No Upfront](AWS_NOtes_Images/EC2_Saving_Plans.png)

![AWS Savings Plans (part 2) — the 3 savings types in detail: Compute Savings Plans provide the most flexibility and reduce costs by up to 66%, automatically applying to EC2 instance usage plus AWS Fargate and AWS Lambda regardless of instance family, size, AZ, region, OS, or tenancy; EC2 Instance Savings Plans provide the lowest prices with savings up to 72% in exchange for committing to a specific instance family in a region, automatically reducing cost on that family regardless of AZ, size, OS, or tenancy and allowing usage to shift between instances within that family in that region; SageMaker Savings Plans reduce SageMaker costs by up to 64% regardless of instance family, size, component, or region](AWS_NOtes_Images/EC2_Saving_Plans_2.png)

**Savings Plans give similar discounts to RIs but simplify the purchasing process:** instead of reserving specific instances, you **commit to a consistent amount of spend ($/hour)** for a **1 or 3-year** term (with **All / Partial / No Upfront** options, same as RI).

**The 3 types of Savings Plan:**
| Type | Savings | Applies to |
|---|---|---|
| **Compute Savings Plan** | up to **66%** (**most flexible**) | **EC2 + Fargate + Lambda**, regardless of **family, size, AZ, Region, OS, or tenancy** |
| **EC2 Instance Savings Plan** | up to **72%** (**lowest prices**) | A **specific instance family in a specific Region** — flexible on AZ, size, OS, tenancy **within that family/region** |
| **SageMaker Savings Plan** | up to **64%** | **SageMaker** usage, regardless of family/size/component/Region |

> **RI vs Savings Plan (the trade-off):** RIs can give a slightly higher max discount (up to 75% Standard) **and** can reserve capacity (Zonal) + be sold on the Marketplace. **Savings Plans are simpler to buy and more flexible** (Compute SP even covers Fargate & Lambda) but **don't reserve capacity** and **can't be resold.** Modern AWS guidance leans toward **Savings Plans** for most steady compute.

---

### Decision cheat — which model do I pick?

| Workload pattern | Best model |
|---|---|
| Unpredictable / spiky / short-term / new app | **On-Demand** |
| Steady, 24×7, predictable for 1–3 years | **Reserved Instances** or **Savings Plans** |
| Need a **guaranteed capacity** reservation in an AZ | **Zonal Reserved Instance** |
| Want flexible discount across AZs/sizes/Fargate/Lambda | **Compute Savings Plan** |
| Fault-tolerant, interruptible, batch / big data | **Spot** |
| Isolated hardware / BYOL / compliance | **Dedicated** (Host) |

> **One-line summary:** **On-Demand** = flexible, no commit, baseline price. **Reserved/Savings Plans** = commit 1–3 yrs for **up to ~72–75% off** steady workloads (RI can also **reserve capacity**; Savings Plans are **simpler & broader**). **Spot** = **up to 90% off** but **interruptible** (batch/fault-tolerant only). **Dedicated** = **priciest**, isolated hardware for compliance/licensing.

## 13. Common Confusions to Nail

1. **Memory Optimized vs Storage Optimized.** Memory Optimized = data processed **in RAM** (in-memory DB/cache). Storage Optimized = data on **fast local disk** (data warehousing, big sequential I/O). The word **"in-memory" → Memory**; **"warehouse / large datasets on disk" → Storage**.
2. **Compute Optimized vs General Purpose.** General Purpose is **balanced** (default choice). Reach for Compute Optimized only when the workload is clearly **CPU-bound** (high-performance processor needed).
3. **Accelerated ≠ Compute Optimized.** "High performance" can trick you — if it mentions **GPU, ML, FPGA, graphics, inference**, it's **Accelerated**, not Compute Optimized.
4. **Instance vs AMI vs Instance Type.** **AMI** = the *image/template* (OS + software). **Instance type** = the *hardware size/family* (e.g. `m5.large`). **Instance** = the *running VM* you get after launching an AMI on an instance type.
6. **Family vs Size.** In `c5.xlarge`, the **family** (`c5`) decides *which kind* of hardware (CPU-optimized); the **size** (`xlarge`) decides *how much* of it. Don't confuse "type of resource" with "amount of resource."
7. **Bigger size ≠ different family.** Going `large → 2xlarge` only **scales capacity up** — it does **not** change the family's strength. Switch the **letter** to change the *kind* of optimization.
8. **Dedicated Instance vs Dedicated Host.** Both are single-tenant. **Dedicated Instance** = isolated hardware, **no host visibility, no BYOL**. **Dedicated Host** = you get the **whole physical server** with socket/core visibility → the **only one that supports BYOL** and physical-placement control. "License" → **Host**.
9. **BYOL is *not* a separate tenancy.** It's a *reason* to use a **Dedicated Host**. Don't pick "Dedicated Instance" for licensing — it can't see the physical sockets/cores the license needs.
5. **EC2 is IaaS, so the OS is yours.** AWS gives you the VM; **you** patch and secure the guest OS, firewall (Security Group), and apps — see [[04_Shared_Responsibility_Model]].
10. **Public IP vs Elastic IP.** A default **public IPv4** is *dynamic* — it **changes** on stop/start. An **Elastic IP** is *static* and stays the same. If a question needs a **fixed** address, it's an **EIP**.
11. **Elastic IP billing is backwards from intuition.** It's **free while attached to a running instance** and **charged when idle/unattached.** "Charged for an EIP" → it isn't in use.
12. **AMI vs Launch Template.** Both feel like "templates" but differ: **AMI = the disk image** (OS + software). **Launch Template = the saved launch settings** (which AMI, instance type, key pair, SG…). The Launch Template **points at** an AMI.
13. **AMI vs EBS Snapshot.** A **snapshot** backs up **one volume**; an **AMI** is a full **launchable template** (OS + permissions + one-or-more snapshots). You **launch an instance** from an AMI; you **restore a volume** from a snapshot.
14. **Launch Template vs Launch Configuration.** **Launch Template** is **newer, supports versioning**, and is **recommended**. **Launch Configuration** is **legacy, no versioning.** New setups → **Launch Template**.
15. **ALB vs NLB (the #1 load-balancer trap).** **ALB = Layer 7**, understands **HTTP/HTTPS**, routes by **URL/path/host** → web apps & APIs. **NLB = Layer 4**, raw **TCP/UDP**, **fastest**, and the **only one with a static/Elastic IP** → high performance, non-HTTP, or partner IP allow-listing. "Route by URL" → ALB; "millions of req/s or static IP" → NLB.
16. **ELB scales itself — you don't.** It's managed and auto-scaling; its IPs churn as it grows, which is why you advertise the **DNS name** (ALB) or the **fixed Elastic IP** (NLB), never a raw ALB IP.
17. **Load Balancer ≠ Auto Scaling.** The **ELB spreads traffic and hides failures**; **Auto Scaling changes how many instances exist.** They're partners, not the same thing — exam loves pairing them for "highly available + scalable."
18. **Health check vs Sticky session.** A **health check** decides *which instances are alive*; a **sticky session** decides *which alive instance a given user keeps hitting*. Different jobs.
19. **Target Group vs Instance.** Targets are registered in a **target group** (instances, IPs, containers, or Lambda for ALB); the LB's **listener rules** forward to target groups. You don't point a listener straight at an instance.
20. **ASG vs ELB (don't merge them).** The **ASG decides *how many* instances** (launch/replace/scale); the **ELB decides *where each request* goes**. The ASG **registers its instances into the Target Group** that the ELB serves — that's the link between them.
21. **Desired vs Min vs Max.** **Desired** = what the ASG maintains *now*; **Min** = floor it won't go below; **Max** = ceiling it won't exceed. Scaling policies move **Desired** between Min and Max.
22. **EC2 health check vs ELB health check (on the ASG).** **EC2** check = "is the VM running?" (misses a crashed app). **ELB** check = "does the app respond 200?" Use **ELB health checks** so a broken *application* gets replaced, not just a dead VM.
23. **Route 53 ≠ load balancing.** Route 53 is **DNS** (find the front door) and can do failover/latency/weighted/geo routing + health checks — but it does **not** balance per-request load across instances. The **ELB** does that.
24. **The full HA stack.** **Route 53 → ELB → Target Group → ASG (using a Launch Template) → instances across multiple AZs.** Know which box owns which job — the exam tests "which service do you add to get HA/elasticity/self-healing?"
25. **Spot ≠ "cheap On-Demand."** Spot is up to **90% off** but **AWS can reclaim it any time** — only for **interruptible/fault-tolerant** work. Never put a database or a single non-checkpointed job on Spot.
26. **Spot billing direction.** **AWS** terminates → you're **not** charged the partial hour. **You** terminate → you **are** charged for the hour it ran. (Easy to get backwards.)
27. **Standard vs Convertible RI.** **Standard** = higher discount (75%), can **modify** but **not change family**. **Convertible** = lower discount (54%), can **exchange** for a different family/OS (equal-or-greater value). "Need flexibility to change family" → **Convertible**.
28. **Regional vs Zonal RI.** **Regional** = billing discount only, **floats across AZs + instance sizes** (size flexibility), **no capacity guarantee**. **Zonal** = locked to one AZ + size but **reserves capacity**. "Guaranteed capacity in an AZ" → **Zonal**.
29. **RI vs Savings Plan.** Both commit 1–3 yrs for big savings. **RI** can **reserve capacity** (Zonal) + be **resold** (Marketplace). **Savings Plan** is **simpler to buy, more flexible** (Compute SP covers **Fargate & Lambda** too) but **no capacity reservation / no resale**.
30. **Reserved Instance ≠ a different instance.** An RI is a **billing discount + (optionally) a capacity reservation** applied to matching On-Demand usage — it's **not** a separate type of VM you launch.

### Top exam points to remember
1. **EC2 = virtual machines (servers) in the cloud**; a launched VM is an **instance**, built from an **AMI** (its OS+software template).
2. **EC2 is the backbone of AWS** — S3, RDS, DynamoDB, Lambda all run on EC2 underneath.
3. **5 instance families** map to the **scarce resource**: **General Purpose** (balanced), **Compute** (CPU), **Memory** (RAM/in-memory), **Accelerated** (GPU/FPGA/ML), **Storage** (local disk I/O).
4. **Match the keyword to the family** — "in-memory" → Memory, "ML/GPU" → Accelerated, "data warehouse" → Storage, "CPU-intensive" → Compute, "web server" → General Purpose.
5. **EC2 is IaaS** → **you** manage and patch the OS; AWS manages the hypervisor and physical hardware.
6. **An instance type = family + generation + size** (e.g. `t3.micro` = General Purpose, gen 3, micro). **Family = which hardware, size = how much.** Each size step (nano → micro → … → xlarge → 2xlarge) **roughly doubles capacity and cost**; **`t2.micro` is the Free Tier** instance.
7. **Tenancy:** **Shared** (default, cheapest) → **Dedicated Instance** (single-tenant hardware, for "no shared hardware" compliance) → **Dedicated Host** (your **whole physical server**, the **only** option that supports **BYOL**/per-socket licensing and physical placement). **"Bring Your Own License" ⇒ Dedicated Host.**
8. **Elastic IP = static public IPv4** that **doesn't change** on stop/start and can be **remapped** for failover. **Free while attached to a running instance; charged when idle** — "billed for an EIP" means it's allocated but not in use.
9. **AMI = a launchable template** (OS + software + EBS snapshots + permissions). 4 sources: **Quick Start (AWS), Marketplace (3rd-party), My AMIs (custom golden image), Community.** AMIs are **Region-scoped** → **copy** to use cross-Region.
10. **Launch Template = saved, versioned launch settings** (which AMI, instance type, key pair, SG, user data) for consistent/repeatable launches and Auto Scaling. It's the **recommended** choice; **Launch Configuration is the legacy, no-versioning** predecessor. Don't confuse the **AMI** (the image) with the **Launch Template** (the launch settings that point at an AMI).
11. **ELB = managed load balancer** that gives you **one stable front door** over many instances: it **distributes traffic**, runs **health checks** (auto-removing dead instances), spans **multiple AZs**, and **scales itself**. It's the standard **HA + scalability** building block alongside **Auto Scaling**.
12. **Know the 4 ELB types:** **ALB** (Layer 7 — HTTP/HTTPS, routes by URL/path/host, for web apps/APIs/containers, **DNS name**), **NLB** (Layer 4 — TCP/UDP, **fastest**, the **only one with a static/Elastic IP**, for high-throughput & non-HTTP), **GLB** (Layer 3 — routes through 3rd-party security appliances), **CLB** (legacy). **"Route by URL" → ALB; "millions of requests / static IP" → NLB.**
13. **Advertise the right endpoint:** for an **ALB** give out its **DNS name** (CNAME / Route 53 Alias) — its IP isn't static. For an **NLB** you can hand out a **fixed Elastic IP** (e.g. for partner allow-listing). Never publish a raw ALB IP.
14. **Core LB building blocks:** **Listener** (port/protocol) → **Listener rules** → **Target Group** (the backend pool) → **Health checks** decide who's alive. Extras devs use: **sticky sessions** (session affinity), **SSL/TLS termination** (+ ACM certs), **cross-zone load balancing**, and **internet-facing vs internal**.
15. **Auto Scaling Group (ASG) = elasticity + self-healing.** It **adds/removes** instances to match demand and **auto-replaces** any that fail a health check, keeping the count at **Desired** (between **Min** and **Max**). It launches them from a **Launch Template** and spreads them across **multiple AZs**.
16. **Scaling policies:** **Target Tracking** (hold a metric, e.g. CPU 50% — the go-to), **Step/Simple** (alarm-driven adjustments), **Scheduled** (known times), **Predictive** (ML forecast). Triggers come from **CloudWatch** alarms.
17. **Use ELB health checks on the ASG** so a crashed *application* (not just a dead VM) gets replaced — the **EC2** check only sees whether the VM is running.
18. **Connect the dots (the #1 architecture):** **Route 53** (DNS) → **ELB** (distributes requests) → **Target Group** (backend pool + health checks) → **ASG** (right-sizes & self-heals the fleet via a **Launch Template**) → **instances across multiple AZs**. The ASG **registers instances *into* the target group**; the ELB routes *to* that target group. Know **which component owns which job** — that mapping is exactly what scenario questions test.
19. **5 EC2 pricing models + commitment = discount.** **On-Demand** (no commit, default, spiky/unpredictable) → **Reserved/Savings Plans** (commit 1–3 yr, **up to ~72–75% off** steady usage) → **Spot** (**up to 90% off** but **interruptible** — batch/fault-tolerant only) → **Dedicated** (priciest, isolated hardware for compliance/BYOL). Match the **workload pattern** to the model.
20. **Reserved Instance detail:** **Standard** (75%, modify only) vs **Convertible** (54%, exchange family/OS); **All/Partial/No Upfront** (more upfront = bigger discount); **Regional RI** = flexible discount across AZs/sizes with **no capacity guarantee**, **Zonal RI** = **reserves capacity** in one AZ. RIs can be **shared across an AWS Organization** and **sold on the RI Marketplace**. Default limits: **20 Regional RIs/Region + 20 Zonal RIs/AZ per month**.
21. **Spot termination billing:** if **AWS** kills it you **aren't** charged the partial hour; if **you** kill it you **are** charged the hour it ran. **AWS Batch** is the easy way to run on Spot.
22. **RI vs Savings Plan:** Savings Plans **simplify buying** (commit **$/hr**) and are **more flexible** — **Compute SP** (up to 66%) even covers **Fargate & Lambda**; **EC2 Instance SP** (up to 72%) locks to one **family in a Region**. But only **RIs reserve capacity (Zonal) and can be resold**.
